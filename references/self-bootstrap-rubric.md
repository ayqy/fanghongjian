# 方法系统自举诊断 rubric

## 1. 什么时候必须用这个文件

当分析对象不是业务问题，而是当前 `skill / prompt / workflow / rubric / taxonomy / eval` 本身时，立即使用这个文件。

典型信号：

- 用户在问“为什么这个系统不会泛化”
- 用户在问“为什么它总掉回修案例”
- 用户在问“该怎么改这个 skill / prompt / workflow 自己”
- 用户在问“为什么 eval 一换说法就失效”

## 2. 先锁主专业角色

这类问题的主专业角色不是任何失败案例里的一线角色，而是系统 owner。

这个 owner 至少守住三件事：

1. 抽象逻辑
2. 选择规则
3. 评测 / 回归门

如果答案开头还在谈医护、育儿、内容、数据等案例 owner，直接判偏题。

## 3. 五层必答诊断

每次都要按下面五层诊断，缺一层都不算答到位：

1. 问题抽象层
   - 系统现在依据什么读出真正约束
   - 这里有没有把表象词误当成真正问题
2. 角色生成层
   - 系统现在依据什么长出主专业角色
   - 有没有偷偷借旧案例角色名做路由
3. 控制面选择层
   - 系统现在依据什么长出方法论主线
   - 有没有把案例里的动作清单当成方法论
4. 场景翻译层
   - 系统如何把通用方法论压回当前场景
   - 有没有把翻译层反客为主，重新变成案例本体
5. 评测 / 回归层
   - 系统用什么证明自己真的泛化
   - 有没有只在老样本或固定问法上看起来正常

## 4. pass / fail 条件

### 直接 fail

- 在给出系统 owner 之前，大段展开具体失败 case
- 在五层诊断之前，先给“再补几个案例”“把这两个例子写细一点”之类建议
- 在前四段里点名医护、育儿、内容、数据等旧业务案例，并把它们当论证主线
- 把例子继续当运行时逻辑，而不是回归输入
- 只说理念正确，没有回归门

### 最低 pass

- 明确系统 owner
- 明确五层里至少哪一层失效、为什么失效、怎么改
- 明确回归门如何证明系统不再靠案例活着
- 明确案例只作为回归输入，不再承担选择逻辑
- 在前四段里不让旧业务案例反客为主

## 5. 重写顺序

如果需要真的改系统，默认按下面顺序动：

1. 先改入口路由
   - 哪些词、对象或提问形态一出现，就必须切到元问题分支
2. 再改五层生成逻辑
   - 抽象
   - 选角
   - 选控制面
   - 翻译
   - 回归
3. 再改输出契约
   - 保证系统 owner、五层诊断、回归 gate 先于案例修法
4. 最后才改样例和措辞
   - 样例只做回归输入和命名校准

## 6. meta eval 怎么设计

不要只放一条固定问法。meta eval 至少覆盖下面变化：

- `skill` / `prompt` / `workflow` / `rubric` / `taxonomy` / `eval` 不同对象名
- “不会泛化”“总掉回修案例”“一换说法就失效”“靠白名单活着”等不同抱怨方式
- 直接点名旧问题，和完全不点名旧问题，两种表述
- 正式提问、口语化吐槽、审稿式反馈三种语气

判断标准不是“命中某个关键词”，而是：

- 是否稳定先说系统 owner
- 是否稳定进入五层诊断
- 是否稳定给出跨样本回归 gate
- 是否稳定把旧业务案例压到后半段，只作为回归输入

## 7. 一句话验收

如果看完答案，用户仍然觉得“你只是换个方式继续修两个 case”，就说明这次自举诊断失败。
