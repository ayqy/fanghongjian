# 专业角色锚点法

## 为什么先锚定角色

同一个问题，换一个专业角色看，方法论会完全不同。

- 写作者会先想怎么写
- 编辑会先想怎么改
- 产品经理会先想目标和验收
- LLM 应用工程师会先想任务定义、评测集和变量控制

所以不要直接从“问题像什么”跳到“我给你几个办法”；要先回答：

`在这个问题上，谁才是真正的专业角色？`

## 1. 先选主专业角色

按下面顺序判断：

1. 如果这件事失败，谁最应该对结果负责
2. 谁在定义“什么叫做好”
3. 谁拥有最关键的产物、决策或验收门
4. 谁的前三个问题最能决定后续顺序

满足越多，越接近主专业角色。

如果涉及多人协作：

- 只选 1 个主专业角色
- 其他角色标成协作方、约束方或依赖方

如果当前领域没有现成头衔，也不要退回 `PM / 协调人 / 项目负责人`。先写成职责型角色：

- `对结果负责的人`
- `定义合格线的人`
- `守住不可逆风险的人`
- `拥有关键交付门的人`

然后再把它翻译成更贴近行业母语的角色名。

## 2. 再从角色职能提炼方法论

角色不是标签，关键是职能。

至少回答三件事：

1. 这个角色的核心职责是什么
2. 这个角色平时守住什么指标、规则或交付物
3. 这个角色遇到同类问题时，第一反应会先定什么、先看什么、先排什么顺序

如果说不出这三件事，说明角色锚点还不够实。

## 3. 对外输出模板

优先使用下面两句：

- `在这个问题上，真正专业的不是 A，而是 B；因为 B 的职责不是泛泛处理，而是守住 C。`
- `所以方法论不是“多试几次”，而是 B 角色常用的：先 D，再 E，最后用 F 验收。`

## 4. 常见角色锚点样例（不是白名单）

下面这些只是高频样例，目的是示范“角色应该长成什么样”，不是穷举列表。陌生领域优先沿用上面的判断规则长出领域内角色，再参考这里的写法校准“像不像内行的母语”。

### 数据 / 指标争议

- 主专业角色：`数据质量负责人 / 分析工程负责人`
- 核心职责：定义唯一口径、建立仲裁机制、证明数据在什么条件下算真

### 跨部门台账 / 状态混乱

- 主专业角色：`运营治理负责人 / PMO`
- 核心职责：统一状态定义、统一节奏、让台账能支持决策而不是制造噪音

### 基层公共事务协调

- 主专业角色：`政策执行协调负责人 / 基层事务推进负责人`
- 核心职责：让程序、补偿、异议处理和推进节奏同时成立

### Prompt 优化 / 模型输出质量

- 主专业角色：`LLM 应用工程师 / Prompt 工程师 / Evaluation 工程师`
- 核心职责：
  - 先把任务定义清楚
  - 先把好坏样本建出来
  - 先把错误类型拆开
  - 先做单变量试验
  - 先设回归门，再扩大使用

对这类问题，默认不要把主专业角色写成“写作者”或“编辑”，除非用户真正的问题是人工写作，而不是模型调优。

### Skill / Prompt / Taxonomy 架构

- 主专业角色：`知识系统架构负责人 / Prompt 架构负责人 / Evaluation 负责人`
- 核心职责：
  - 先分清系统真正靠什么泛化
  - 先把不变量和样例分层
  - 先定义回归门，再谈局部补丁
  - 先保证陌生样本也能长出正确角色和方法论

对这类问题，默认不要把主专业角色写成某个失败 case 里的一线角色；如果分析对象本身就是 skill / prompt / taxonomy，主专业角色应先指向系统所有者，而不是案例所有者。

## 5. 领域内角色生成法

遇到医护、育儿、法务、研究、采购、设计等陌生领域时，不要先想“它像哪个通用岗位”，而要先写出下面四行：

1. `结果一旦出事，谁先负责`
2. `什么叫合格，由谁定义`
3. `最不能错的风险是什么，由谁守住`
4. `关键交付门或升级门在谁手里`

然后把四行压成一个角色句：

- `在这个问题上，主专业角色是 [角色名]；他的职责不是泛泛协调，而是守住 [结果 / 风险 / 验收门]。`

如果行业里没有统一头衔，就先用职责型角色，再尽量贴近行业母语。判断标准不是“名词高级”，而是“内行听完会点头，知道这确实是那个抓法”。

### 医护执行分歧示例

- 不要直接写：`协调负责人`
- 更像内行的写法：`病区当班责任护士 / 护士长 / 用药安全负责人`
- 因为这个问题首先不是“谁去沟通”，而是“谁有责任在信息冲突时先守住用药安全和执行边界”

### 育儿作息失控示例

- 不要直接写：`家庭项目负责人`
- 更像内行的写法：`主要照护者 / 家庭作息设计负责人 / 儿童睡眠支持负责人`
- 因为这个问题首先不是“谁来管大家”，而是“谁来统一作息规则、照护动作和哭闹升级阈值”

### 当问题对象就是当前方法系统

- 不要直接写：`医护负责人 / 育儿负责人 / 某个失败 case 的一线角色`
- 更像内行的写法：`知识系统架构负责人 / Prompt 架构负责人 / Evaluation 负责人`
- 因为这类问题首先不是“那两个案例该怎么解”，而是“这个系统为什么会把例子误当成生成机制，以及应该怎样重写选择规则和回归门”

## 6. 自检标准

如果你已经写完答案，再回头检查：

1. 用户能不能一眼看出“这件事该按谁的工作抓法来做”
2. 方法论是不是从这个角色的职责自然长出来的
3. 删掉角色名称后，方法论会不会失去依据
4. 当前场景如果有明显行业风险，角色名称有没有贴近那个行业的实际守门人
5. 如果答案只剩 `协调 / 推进 / 统一 / 沟通` 这些词，是否说明你还没找到真角色
6. 如果问题本身是在说“这个系统不会泛化”，你是否仍然把角色错锚到了失败案例，而不是系统所有者

如果第 3 条答案是“会”，说明角色锚点有效。
