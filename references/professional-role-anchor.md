# 专业角色锚点法

## 为什么先锚定角色

同一个问题，换一个专业角色看，方法论会完全不同。

- 写作者会先想怎么写
- 编辑会先想怎么改
- 产品经理会先想目标和验收
- LLM 应用工程师会先想任务定义、评测集和变量控制

所以不要直接从“问题像什么”跳到“我给你几个办法”；要先回答：

`在这个问题上，谁才是真正的专业角色？`

## 1. 先选主专业角色

按下面顺序判断：

1. 如果这件事失败，谁最应该对结果负责
2. 谁在定义“什么叫做好”
3. 谁拥有最关键的产物、决策或验收门
4. 谁的前三个问题最能决定后续顺序

满足越多，越接近主专业角色。

如果涉及多人协作：

- 只选 1 个主专业角色
- 其他角色标成协作方、约束方或依赖方

## 2. 再从角色职能提炼方法论

角色不是标签，关键是职能。

至少回答三件事：

1. 这个角色的核心职责是什么
2. 这个角色平时守住什么指标、规则或交付物
3. 这个角色遇到同类问题时，第一反应会先定什么、先看什么、先排什么顺序

如果说不出这三件事，说明角色锚点还不够实。

## 3. 对外输出模板

优先使用下面两句：

- `在这个问题上，真正专业的不是 A，而是 B；因为 B 的职责不是泛泛处理，而是守住 C。`
- `所以方法论不是“多试几次”，而是 B 角色常用的：先 D，再 E，最后用 F 验收。`

## 4. 常见角色锚点

### 数据 / 指标争议

- 主专业角色：`数据质量负责人 / 分析工程负责人`
- 核心职责：定义唯一口径、建立仲裁机制、证明数据在什么条件下算真

### 跨部门台账 / 状态混乱

- 主专业角色：`运营治理负责人 / PMO`
- 核心职责：统一状态定义、统一节奏、让台账能支持决策而不是制造噪音

### 基层公共事务协调

- 主专业角色：`政策执行协调负责人 / 基层事务推进负责人`
- 核心职责：让程序、补偿、异议处理和推进节奏同时成立

### Prompt 优化 / 模型输出质量

- 主专业角色：`LLM 应用工程师 / Prompt 工程师 / Evaluation 工程师`
- 核心职责：
  - 先把任务定义清楚
  - 先把好坏样本建出来
  - 先把错误类型拆开
  - 先做单变量试验
  - 先设回归门，再扩大使用

对这类问题，默认不要把主专业角色写成“写作者”或“编辑”，除非用户真正的问题是人工写作，而不是模型调优。

## 5. 自检标准

如果你已经写完答案，再回头检查：

1. 用户能不能一眼看出“这件事该按谁的工作抓法来做”
2. 方法论是不是从这个角色的职责自然长出来的
3. 删掉角色名称后，方法论会不会失去依据

如果第 3 条答案是“会”，说明角色锚点有效。
