# 专业流程原型地图

遇到陌生问题时，先判断它属于哪一类，再判断“谁是这件事的主专业角色”，然后再套用对应的专业流程原型。不要在这里停住；选完原型后，还要回到 `references/philosophical-elevation.md` 做一次内部校准，再把结果翻成一句更朴素的方法论主线，而不是直接输出高层术语。

这里列出的角色名称都只是常见样例，不是白名单。

如果当前领域不在样例里，先用下面四个问题生成一个“领域内主专业角色”，再来套流程原型：

1. 如果结果出事，谁最先承担结果责任
2. 谁在定义这件事什么叫“安全”或“合格”
3. 谁守着最不能出错的不可逆风险
4. 谁拥有最关键的产物、交接门或验收门

流程原型可以复用，但角色名称应尽量贴近行业母语；不要为了迁就样例表，把医护、育儿、研究、法务等问题强行写成 `PM / 协调负责人 / 项目经理`。

## 1. 故障排查类

- 典型场景：系统出错、指标异常、交付失败、线上事故、流程失灵
- 常见主专业角色样例：`故障负责人 / SRE / 数据质量负责人 / 分析工程负责人`
- 核心职能：固定现场、建立唯一事实、定位根因、给出止血与复发控制
- 内行流程：
  1. 固定现场并确认影响范围
  2. 按时间线和依赖链排查
  3. 区分根因、诱因、放大器
  4. 先止血，再修复，再复盘
- 外行控制点：
  - 影响范围是否说清
  - 止血动作是否先于长期修复
  - 验证方式是否能证明问题已消失而不是暂时没复现

## 2. 构建交付类

- 典型场景：做产品、做功能、做报告、做系统、做活动、做内容
- 常见主专业角色样例：`产品负责人 / 解决方案负责人 / 内容负责人 / 项目负责人`
- 核心职能：定义目标、定义验收、拆阶段、控依赖、把中间结果推到正式交付
- 内行流程：
  1. 锁定目标和验收标准
  2. 拆依赖、拆阶段、拆责任
  3. 先出最小可验证结果
  4. 按反馈迭代到正式交付
- 外行控制点：
  - 验收标准是否早于执行
  - 依赖和风险是否前置暴露
  - 是否存在可演示的中间产物

## 3. 选择评估类

- 典型场景：选方案、选产品、选供应商、选市场、选技术路线
- 常见主专业角色样例：`评估负责人 / 技术负责人 / 采购负责人 / 研究负责人`
- 核心职能：定义标准、拉齐约束、比较证据、做取舍并承担推荐责任
- 内行流程：
  1. 先定评估维度和权重
  2. 拉齐备选项与约束
  3. 做证据对照和成本测算
  4. 给出推荐与放弃理由
- 外行控制点：
  - 标准是否在结论前就已固定
  - 是否把不能接受的底线单独列出
  - 推荐理由是否能回指到证据

## 4. 运营救火类

- 典型场景：项目失控、团队扯皮、进度拖延、质量反复、客户升级
- 常见主专业角色样例：`运营负责人 / PMO / 项目救火负责人`
- 核心职能：重建真实进度、抓主路径、收紧节奏、恢复稳定
- 内行流程：
  1. 重建真实进度和问题清单
  2. 区分主路径问题与噪音
  3. 收紧角色、节奏、升级路径
  4. 先恢复稳定，再优化效率
- 外行控制点：
  - 有没有唯一主路径
  - 谁能拍板、谁负责执行、谁验收是否清楚
  - 升级和回退条件是否明示

## 5. 协同沟通类

- 典型场景：跨团队合作、需求传达、专家沟通、向上汇报、对外协调
- 常见主专业角色样例：`项目协调负责人 / 秘书长角色 / 政策执行协调人 / PM`
- 核心职能：对齐目标、对齐事实、对齐动作、对齐升级条件
- 内行流程：
  1. 对齐目标与边界
  2. 对齐事实与分歧
  3. 对齐下一步和验收
  4. 对齐风险和升级条件
- 外行控制点：
  - 会议或沟通后是否留下了可检查的动作项
  - 分歧是事实分歧还是立场分歧
  - 责任和截止时间是否明确

## 6. 照护支持类

- 典型场景：医护执行分歧、康复照护、睡眠作息混乱、育儿支持、老人照护、个案管理
- 常见主专业角色样例：`病区当班负责人 / 护士长 / 用药安全负责人 / 主要照护者 / 家庭作息设计负责人 / 儿童发展支持负责人`
- 核心职能：先守住对象安全与节律稳定，再统一照护规则、观察窗口、升级条件和回退边界
- 内行流程：
  1. 先冻结权威指令、作息基线或照护规则
  2. 先区分安全风险、执行偏差和偏好冲突
  3. 统一照护动作、例外条件和升级触发器
  4. 以小步试运行观察反应，再决定扩大、维持或回退
  5. 出现红旗时优先升级给具备专业处置权限的人
- 外行控制点：
  - 权威版本是否唯一，例如最新医嘱、统一作息表、统一照护口径
  - 执行差异和偏好冲突是否被拆开，而不是混成“大家都不一致”
  - 升级触发器是否明确，例如异常疼痛、用药冲突、持续失眠、情绪失控、身体红旗
  - 观察窗口和回退条件是否写清，而不是只说“再看看”

## 7. LLM 调优类

- 典型场景：提示词优化、模型输出质量不稳、回答像模板、风格不一致、自动化链路效果差
- 常见主专业角色样例：`LLM 应用工程师 / Prompt 工程师 / Evaluation 工程师`
- 核心职能：把任务定义清楚、建立评测样本、拆错误类型、做单变量试验、设回归门，最终稳定提升模型表现
- 内行流程：
  1. 先定义任务目标、禁区和验收指标
  2. 先建最小评测集，再拆错误类型
  3. 每次只改一个变量做对照试验
  4. 用样本结果而不是主观感觉筛 prompt
  5. 固化最优链路并建立回归 gate
- 外行控制点：
  - 是否先有评测题集，再开始调 prompt
  - 每次改动是不是只改一个变量
  - 是否有明确错误分类，而不是笼统说“效果不好”
  - 是否有回归门，能证明新版本比旧版本更稳

## 8. 默认输出骨架

如果一时不能确定具体行业，至少输出：

1. 当前问题属于哪一类原型
2. 这件事上谁是主专业角色，他的职责是什么；如果没有现成头衔，也要输出职责型角色
3. 如果按这个角色的职能来做，这类问题真正该先立什么、再推进什么、最后如何验收
4. 该原型下最专业的标准流程是什么
5. 当前场景优先要抓的三个控制点是什么
6. 哪些地方是直接证据，哪些地方只是推断
