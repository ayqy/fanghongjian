# 专业流程原型地图

遇到陌生问题时，先判断它属于哪一类，再套用对应的专业流程原型。

## 1. 故障排查类

- 典型场景：系统出错、指标异常、交付失败、线上事故、流程失灵
- 内行流程：
  1. 固定现场并确认影响范围
  2. 按时间线和依赖链排查
  3. 区分根因、诱因、放大器
  4. 先止血，再修复，再复盘
- 外行控制点：
  - 影响范围是否说清
  - 止血动作是否先于长期修复
  - 验证方式是否能证明问题已消失而不是暂时没复现

## 2. 构建交付类

- 典型场景：做产品、做功能、做报告、做系统、做活动、做内容
- 内行流程：
  1. 锁定目标和验收标准
  2. 拆依赖、拆阶段、拆责任
  3. 先出最小可验证结果
  4. 按反馈迭代到正式交付
- 外行控制点：
  - 验收标准是否早于执行
  - 依赖和风险是否前置暴露
  - 是否存在可演示的中间产物

## 3. 选择评估类

- 典型场景：选方案、选产品、选供应商、选市场、选技术路线
- 内行流程：
  1. 先定评估维度和权重
  2. 拉齐备选项与约束
  3. 做证据对照和成本测算
  4. 给出推荐与放弃理由
- 外行控制点：
  - 标准是否在结论前就已固定
  - 是否把不能接受的底线单独列出
  - 推荐理由是否能回指到证据

## 4. 运营救火类

- 典型场景：项目失控、团队扯皮、进度拖延、质量反复、客户升级
- 内行流程：
  1. 重建真实进度和问题清单
  2. 区分主路径问题与噪音
  3. 收紧角色、节奏、升级路径
  4. 先恢复稳定，再优化效率
- 外行控制点：
  - 有没有唯一主路径
  - 谁能拍板、谁负责执行、谁验收是否清楚
  - 升级和回退条件是否明示

## 5. 协同沟通类

- 典型场景：跨团队合作、需求传达、专家沟通、向上汇报、对外协调
- 内行流程：
  1. 对齐目标与边界
  2. 对齐事实与分歧
  3. 对齐下一步和验收
  4. 对齐风险和升级条件
- 外行控制点：
  - 会议或沟通后是否留下了可检查的动作项
  - 分歧是事实分歧还是立场分歧
  - 责任和截止时间是否明确

## 6. 默认输出骨架

如果一时不能确定具体行业，至少输出：

1. 当前问题属于哪一类原型
2. 该原型下最专业的标准流程是什么
3. 用户作为外行最该抓的三个控制点是什么
4. 哪些地方是直接证据，哪些地方只是推断
